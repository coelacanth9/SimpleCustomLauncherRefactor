## 1. 神ViewModel の解体

### Before
`MainViewModel.kt` 1,358行・87関数・10責務グループ。  
ナビゲーション、課金、広告、ショートカット配置、行操作、ページ操作、クリーンアップ、設定、起動処理、エラー管理がすべて1ファイルに混在。

```
MainViewModel.kt（1,358行）
  ├── ナビゲーション管理
  ├── 課金・広告処理
  ├── プレミアム判定
  ├── ショートカット配置・スワップ
  ├── 行追加・削除・列変更
  ├── ページ追加・削除
  ├── パッケージ削除クリーンアップ
  ├── ショートカット起動
  ├── 設定値取得
  └── エラー管理
```

### After
責務ごとに4つの ViewModel に分割。

| ViewModel | 行数 | 責務 |
|---|---|---|
| `HomeViewModel` | 672行 | ホーム画面操作・ナビゲーション・起動・レイアウト操作 |
| `ShortcutSelectViewModel` | 341行 | ショートカット選択・配置書き込み |
| `SettingsViewModel` | 65行 | 設定画面の購入/広告/バックアップ |
| `PremiumViewModel` | 57行 | 課金・広告の UI 状態のみ保持 |

合計 1,135行（旧比 **−17%**）。さらに各 ViewModel の関数が単一責務に絞られ、読み切れる単位になった。

## 2. ViewModel 間の連携をコールバックから StateFlow に

### Before
ショートカット選択後の結果をコールバックで MainViewModel に戻す設計。  
画面遷移のたびにコールバックラムダが増殖し、引数が爆発していた。

```kotlin
// 旧 SlotEditScreen: 29引数（コールバック11個）
fun SlotEditScreen(
    currentShortcut: ShortcutItem?,
    unplacedShortcuts: List<ShortcutItem>,
    placedShortcuts: List<ShortcutItem>,
    onSelectUnplaced: (ShortcutItem) -> Unit,
    onSelectPlaced: (ShortcutItem) -> Unit,
    onSelectApp: (AppInfo) -> Unit,
    onSelectShortcut: (ShortcutData) -> Unit,
    onClear: () -> Unit,
    onChangeColumns: (Int) -> Unit,
    onChangeTextOnly: (Boolean) -> Unit,
    onChangeColors: (String?, String?) -> Unit,
    onDeleteRow: () -> Unit,
    onWatchAd: () -> Unit,
    onPurchase: () -> Unit,
    // ... 計29引数
)
```

### After
`ShortcutRepository` が `StateFlow<LayoutState>` を1つ保持。  
`HomeViewModel` と `ShortcutSelectViewModel` が同じ Flow を購読するため、書き込めば自動で全画面に反映される。

```kotlin
// ShortcutRepository
val layoutState: StateFlow<LayoutState> = _layoutState.asStateFlow()

// 両 ViewModel が同じ Flow を参照するだけ
val layoutState: StateFlow<LayoutState> = shortcutRepository.layoutState

// 新 ShortcutSelectScreen: 2引数のみ
fun ShortcutSelectScreen(
    viewModel: ShortcutSelectViewModel,
    onBack: () -> Unit
)
```

## 3. SlotEdit 画面の廃止・ShortcutSelect への統合

### Before
「空きスロットへの追加」と「配置済みスロットの編集」が別画面（`ShortcutAddScreen` / `SlotEditScreen`）に分かれており、重複ロジックが多かった。

- `ShortcutAddScreen`: 10引数
- `SlotEditScreen`: 29引数
- 合計 2,176行のファイルに両者が詰まっていた

### After
`ShortcutSelectScreen` に統合。配置先スロットに何があるかを ViewModel が知っているため、「新規配置か編集か」の分岐は内部で自動判定。

- `ShortcutSelectScreen`: `viewModel + onBack` の2引数
- 画面数が 7 → 6 に削減

## 4. ナビゲーションの整理

### Before
`MainScreenState` sealed class に `SlotEdit`（引数4個）が存在。  
`MainActivity` の `when(screenState)` 分岐が肥大化し、各分岐でコールバックを大量に注入していた。

```kotlin
// 旧: SlotEdit が NavDestination に混入
sealed class MainScreenState {
    object Home
    data class ShortcutAdd(pageIndex, row, column)
    data class SlotEdit(pageIndex, row, column, currentShortcut)  // ←廃止
    object Calendar
    object Memo
    object AppSettings
    object AllApps
}
```

### After
`NavDestination` は純粋なデータ（コールバックなし）。`SlotEdit` は廃止。  
`MainActivity` の NavHost は対応表を引くだけの薄い `when` 分岐になった。

```kotlin
sealed class NavDestination {
    object Home
    data class ShortcutSelect(val pageIndex: Int, val row: Int, val column: Int)
    object Settings
    object Calendar
    object Memo
    object AllApps
}
```

## 7. platform 層への分離（Intent 生成の移動）

### Before
`ShortcutItem.toIntent()` が `data/ShortcutModel.kt` に存在。  
ドメインモデル（core）が Android の `Intent` / `Uri` に依存しており、依存方向が逆転していた。

### After
`platform/ShortcutIntent.kt` に移動し、`ShortcutItem` への拡張関数として実装。  
`core` は Android 依存ゼロを保持。

```kotlin
// platform/ShortcutIntent.kt
fun ShortcutItem.toIntent(): Intent? = when (type) { ... }
```

## 8. feature → feature 依存の排除

### Before
各画面が `MainViewModel` を直接受け取り、実質的に全機能に依存していた。  
設定画面・カレンダー画面でも `MainViewModel` の課金メソッドや配置メソッドを参照可能な状態。

### After
- `HomeScreen`: 同パッケージの `HomeViewModel` のみ受け取る（OK）
- `SettingsScreen`: 自前の `SettingsViewModel` + ラムダのみ
- `CalendarScreen / MemoScreen / AllAppsScreen`: `onBack` のみ（または最小限の値）
- `ShortcutSelectScreen`: `ShortcutSelectViewModel + onBack` のみ

feature → feature の import は0件。

### Before
`MainActivity.kt` 568行。ナビゲーション分岐・コールバック注入・ViewModel 管理がすべて Activity に集中。
