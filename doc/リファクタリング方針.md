## 決定した方針

**「純粋ロジック切り出し → それを土台に新規構築」**

- フルリライトの即決はしない。
- in-placeリファクタだけで終わらせるのでもない。
- まず現プロジェクト上でAndroid非依存の純粋ロジックを切り出して資産化し、それを土台に新規プロジェクトを構築する。
- この順序は「作り直し」を否定するものではなく、作り直しを**安全かつ速く**するための準備と位置づける。

## 背景（散らかりの実態）

全コード精査（38ファイル・約13,000行）で判明した問題は「アーキの腐敗」ではなく「組織化されていない」こと。

| 症状 | 実数 |
|---|---|
| 神ViewModel | `MainViewModel` 92関数・10責務グループ。課金とページ管理が相互依存で絡む |
| 巨大画面 | HomeScreen 9 Composable / AppSettingsScreen 11 / ShortcutSelectScreen 27引数 |
| 引数過大 | SlotEditScreen 27引数 / ShortcutAddScreen 10引数。選択系8・プレミアム系5が3画面で重複 |
| パッケージ未整理 | ルート直下に6ファイル（CalendarRepository, CalenderComponents 等） |
| コピペ重複 | 楽天Link判定5箇所・ダイアログ雛形・プレミアム誘導UI 3画面 |
| 命名ミス | `Calender`（ファイル名のみ。クラス/関数は正） |

データ層はRepositoryで分離済み、MVVM + sealed classナビは規模相応に健全。

## 検討した選択肢と結論

### 1. in-placeリファクタのみ
- 効果: パッケージ整理・重複解消・引数過大は確実に消える（体感の散らかりの大半）。
- 限界: 骨格は変わらない（MainViewModelは大きな調整役のまま、HomeScreenはVM密結合のまま、sealed classナビもそのまま）。
- 設計思想・土台への不満があるなら満たされない。

### 2. フルリライト即決
- 「動いてる物を一から書き直すな」の定石は、本プロジェクトでは**前提が逆向きで弱い**：
  - 自作コード（暗黙知が失われない）
  - 約13k行（書き直し可能な規模）
  - **未公開**（クローズドベータのみ実施。実ユーザーへのリグレッション実害なし）
- よって未公開・自作・小規模という条件では作り直しは正当な選択肢。
- ただし本当のコストはリグレッションではなく「時間」と「Android連携の地味な配管の再現」。

### 3. ハイブリッド（採用）
- 書き直す価値が高い部分 = UI/ロジック層（不満の本丸。綺麗に組み直せる）。
- 書き直したくない部分 = 課金(BillingManager)・ピンショートカット・HOME intent検知・権限・楽天Linkの癖。
  「綺麗なアーキ」とほぼ無関係に正しく動かすのが面倒な配管。一から書くとここでハマる。
- → **配管は移植・コアロジックは資産化した上で、UIだけ綺麗に組み直す**。フルリライトの旨味だけ取って配管の再ハマりを避ける。

## 進め方ステップ

1. **純粋ロジックの切り出し候補を洗い出す**（現プロジェクト上・低リスク）
   - 配置/グリッド計算、レイアウト設定計算、楽天Link判定、祝日計算、ShortcutType属性化 など
2. 切り出し作業を通じて「現挙動の棚卸し（＝実質の設計書）」を作る（白紙からでなく現状起点）
3. 切り出し後の見通しを見て「作り直す/in-place継続」を最終判断
4. 作り直す場合のみ、抽出済みロジックを土台に構造設計書・実装ステップ資料を作成

### 切り出さない（＝配管として移植する）もの
`toIntent()`、楽天Linkのインストール判定/アイコン取得、ContentProviderクエリ、SharedPreferences/JSONの read/write、BillingManager、ピンショートカット連携。これらは綺麗なアーキと無関係に「正しく動かすのが面倒」な部分なので、書き直さず移植する。

### 純粋ロジック切り出しの判断基準（追記: 2026-06-02）

「切り出す」条件:
- **コピペ重複がある**（複数箇所に同等ロジック） → C分類がこれに該当
- またはロジックが複雑で単体テストの価値が明確に高い

「新規構築時に回す」条件:
- 1箇所のみに存在する
- 実装が2〜3行（filter/map/groupBy等）で自明
→ このケースでの切り出しは「アーキの体裁を整えるだけ」で実質的な改善なし。A分類はこれに該当。
