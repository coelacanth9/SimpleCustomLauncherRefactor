# アーキテクチャ設計方針

作成: 2026-06-03

## 作成経緯

リファクタリング中に `ShortcutRepository` が肥大化していることが判明した。
CRUD以外のビジネスロジック（空きスロット探索・デフォルトレイアウト生成・行削除等）が Repository に集積しており、
`HomeViewModel` でも LauncherApps / PackageManager を直接呼ぶなど、各層の責務が曖昧になっていた。

原因はUseCase層が存在しないことで、ビジネスロジックの置き場がなく Repository と ViewModel に漏れ出ていた。

この状況を解消し「なぜここに置くのか」を明確にするため、
Googleの推奨するレイヤードアーキテクチャ（UI層 / ドメイン層 / データ層）の標準定義に基づいて
各層の配置基準を定めた。

---

## 層構成

標準のレイヤードアーキテクチャ（UI層 / ドメイン層 / データ層）に基づく。

```
UI層       : feature/ + ui/ + navigation/
ドメイン層  : usecase/ + model/
データ層    : data/ + platform/
```

---

## 各パッケージの配置基準

### データ層

| パッケージ | 配置基準 |
|---|---|
| data/ | データの読み書きのみ（CRUD + StateFlow）。ロジックなし |
| platform/ | Android SDKのラップのみ（Manager / Helper）。ロジックなし |

### ドメイン層

| パッケージ | 配置基準 |
|---|---|
| usecase/ | アプリのビジネス操作を表す。関連する操作は1ファイルにまとめてよい（基準は後述）。UIに依存しない |
| model/ | Android非依存の型定義・ドメインルール |

### UI層

| パッケージ | 配置基準 |
|---|---|
| feature/ | UI状態管理（ViewModel）・表示（Screen）のみ。ビジネスロジックなし |
| ui/ | featureをまたぐ共有Composable・テーマ |
| navigation/ | 画面遷移先の型定義。feature間の直接依存を排除 |

### app/

DI起点・画面切り替えのみ。

---

## UseCase のグループ化基準

1ファイルにまとめる「関連」の基準：

| 基準 | まとめる | 分ける |
|---|---|---|
| 操作対象エンティティ | 同一（例: 行への操作はすべて `RowUseCase`） | 異なる |
| 呼び出し文脈 | 常にセットで呼ばれる、または排他的に呼ばれる | 独立して呼ばれることが多い |
| 状態変更の範囲 | 同じ Repository・同じ StateFlow に影響する | 異なる層・異なる状態に影響する |

判断に迷う場合は **分ける** を選ぶ。後から分割する方が、後からまとめるより簡単なため。

---

## 配置判定

どこに置くか迷ったとき：

```
UIの状態・表示                → feature/
ビジネス操作・アプリのロジック → usecase/
データの読み書き              → data/
Android SDKのラップ           → platform/
Android非依存の型             → model/
```

> **DefaultLayoutConfig の Android 依存について（設計上の妥協）**
>
> `DefaultLayoutConfig` が `R.string.xxx` を参照することはドメイン層に Android フレームワーク依存を持ち込む。
> 正しくは UI 層で文字列を解決して UseCase に渡すか、`platform/` に定義データを置く形になるが、
> このアプリでは以下の理由から `usecase/` への同梱を許容する：
> - デフォルトレイアウトは固定であり、差し替えの想定がない
> - UI 層経由で解決済み文字列を渡す形にすると UseCase の引数が増え、呼び出し側が複雑になる

> **`launchShortcut` の分割方針**
>
> `launchShortcut` はIntent起動とナビゲーションが混在しているため、関数ごと移動はできない。
> - Intent / LauncherApps を使う起動処理 → `ShortcutHelper` に新規メソッドとして追加
> - `NavDestination.Calendar` 等への遷移（CALENDAR / MEMO / SETTINGS / ALL_APPS）→ HomeViewModel に残す

> **CleanupUseCase の依存方針**
>
> UseCase は `Context` / `LauncherApps` を直接受け取らない。
> LauncherApps を使う操作（ピンショートカットのunpin等）は `ShortcutHelper` にメソッドを追加し、
> UseCase は `ShortcutHelper` を呼ぶ形にする。
> ```
> CleanupUseCase
>   └─ ShortcutHelper.unpinShortcuts(...)   ← platform
>   └─ ShortcutRepository.deleteXxx(...)    ← data
> ```

> **依存方向について（設計上の妥協）**
>
> UseCase（ドメイン層）が `ShortcutHelper`（platform/ = データ層）に直接依存することは、
> Clean Architecture の原則（依存は内向き）に反する。
> 正しくは、ドメイン層にインターフェースを定義し platform/ が実装する形になる。
>
> このアプリでは以下の理由からインターフェース導入を見送り、直接依存を許容する：
> - 実装が1つしか存在せず、差し替えの想定がない（テスト用モックも現状不要）
> - `ShortcutHelper` は Hilt で注入されるため、依存関係の把握は容易
> - インターフェース追加による記述量増加に対してメリットが得られない
>
> この妥協を前提として、UseCase が `Context` や `LauncherApps` を直接受け取ることは禁止する。
> Android依存の操作は必ず `ShortcutHelper` にメソッドを追加して委譲すること。

| 箇所 | 問題 | 対処 |
|---|---|---|
| `ShortcutReceiverActivity.saveShortcutInfo()` | `pin_shortcuts` SharedPreferencesに直接書いている | `ShortcutRepository.savePinShortcutInfo()`を使うよう修正 |

#### 新規追加

| 追加内容 | 対象 | 理由 |
|---|---|---|
| `update {}` | ShortcutRepository | UseCase が複数の CRUD を呼ぶ際に通知をバッチ化し、StateFlow の二重更新を防ぐ |

> **`update {}` の設計**
>
> ```kotlin
> // Repository 側
> private val isUpdating = AtomicBoolean(false)
>
> fun update(block: ShortcutRepository.() -> Unit) {
>     isUpdating.set(true)
>     try {
>         block()
>     } finally {
>         isUpdating.set(false)  // 成功・例外どちらでも必ず解放
>     }
>     notifyChange()  // finally の後 = isUpdating が false の状態で呼ぶ
>                     // block() が例外を投げた場合はここに到達しない
> }
>
> private fun notifyChange() {
>     if (isUpdating.get()) return  // バッチ中は抑制
>     _layoutState.value = loadLayoutState()
> }
> ```
>
> **前提**: Repository のすべての書き込みメソッドは末尾で `notifyChange()` を呼ぶ。
> `update {}` 実行中はこれが抑制され、ブロック完了後に1回だけ発火する。
> 新しい書き込みメソッドを追加する場合も必ず `notifyChange()` を末尾で呼ぶこと。
>
> **例外時の挙動**: `block()` 内で例外が発生した場合、`notifyChange()` は呼ばれない。
> SharedPreferences の `apply()` は同期的に例外を投げないため、例外の原因は `block()` 内のロジックバグに限られる。
> ロジックバグに対するリカバリは定義しない（修正することが対処）。
> `isUpdating.set(false)` は `finally` に置くことで、失敗後も以降の CRUD 通知が永久に抑制されるのを防ぐ。
>
> **スレッド安全性**
>
> `isUpdating` は `AtomicBoolean` を使用する。`var Boolean` はスレッド間の可視性が保証されないため不可。
> このアプリでは Repository への書き込みはすべてメインスレッドから行われるため `AtomicBoolean` で十分。
> （ViewModel からの呼び出しも、BroadcastReceiver 経由の `onPackageRemoved` も、Android の仕様上メインスレッドで実行される）
> コルーチンの IO ディスパッチャから書き込みを行う場合はこの前提が崩れるため、その際は `Mutex` に切り替えること。
>
> **suspend 関数との非互換**
>
> `update {}` は同期設計であり、suspend 関数との組み合わせは想定しない。
> `block` の中で suspend 関数を呼ぶと `isUpdating` がスレッドをまたいで不整合になる。
> 現状の Repository 操作は全て同期（SharedPreferences の `apply()`）のためこの制約は問題にならないが、
> 将来 Room 等の非同期ストレージに移行する場合は `update {}` ごと `Mutex` ベースの設計に置き換えること。
>
> UseCase 側は操作に必要な呼び出しを `update {}` でまとめる。
> 毎回同じ3つをセットで呼ぶわけではなく、操作ごとに必要なものだけ記述する。
>
> ```kotlin
> // 例: deleteRow
> repository.update {
>     removePlacement(...)   // ループ
>     deleteShortcut(...)    // ループ（条件付き）
>     saveLayoutConfig(...)
> }
>
> // 例: insertRowAt
> repository.update {
>     shiftPlacementsRow(...)
>     saveLayoutConfig(...)
> }
> ```

> **PlaceShortcutUseCase の設計**
>
> ShortcutSelectViewModel の配置操作（placeApp / placeInternalFeature / placeDeviceSettings / placeContact / placeIntent / swapShortcuts）を集約する UseCase。
>
> 内部に private fun `placeItem(item, pageIndex, rowIndex, columnIndex)` を持ち、
> 各 placeXxx から共通的に呼ばれる。`placeItem` は以下を実行する：
> 1. `updateRowFixedHeight()` — DATE_DISPLAY→56dp / TIME_DISPLAY→80dp のビジネスルールを適用
> 2. `repository.savePlacement()` — 配置を保存
>
> 配置操作は saveShortcut() + saveLayoutConfig()（fixedHeight更新）+ savePlacement() の3回の書き込みを含むため、
> `repository.update {}` でバッチ化する。
>
> **依存**: `ShortcutRepository`

> **EditSlotUseCase の設計**
>
> ショートカットスロットの編集操作を集約する UseCase。
> HomeViewModel と ShortcutSelectViewModel で同一ロジックが重複していたものを統合する。
>
> ```kotlin
> class EditSlotUseCase(private val repository: ShortcutRepository) {
>
>     /** スロットをクリアする。shouldDeleteOnRemove に従いショートカット本体も削除 */
>     fun clearSlot(shortcut: ShortcutItem) {
>         repository.update {
>             removePlacement(shortcut.id)
>             if (shouldDeleteOnRemove(shortcut)) deleteShortcut(shortcut.id)
>         }
>     }
>
>     /** スロットの背景色・文字色を変更する */
>     fun changeColors(pageIndex: Int, row: Int, column: Int, backgroundColor: String?, textColor: String?) {
>         val target = repository.layoutState.value.placements
>             .find { it.pageIndex == pageIndex && it.row == row && it.column == column } ?: return
>         repository.savePlacement(target.copy(backgroundColor = backgroundColor, textColor = textColor))
>     }
> }
> ```
>
> **依存**: `ShortcutRepository`
